`LISTING_STATUS_POLICY` shape (D-14 verbatim — 4 keys matching `Car.status` enum):
```
active:    { buyerBlocked: [],                                    banner: null }
suspended: { buyerBlocked: ['add_to_cart','confirm_booking'],     banner: { titleKey: 'listingBannerSuspendedTitle', bodyKey: 'listingBannerSuspendedBody', severity: 'warning' } }
archived:  { buyerBlocked: ['add_to_cart','confirm_booking'],     banner: { titleKey: 'listingBannerArchivedTitle',  bodyKey: 'listingBannerArchivedBody',  severity: 'neutral' } }
deleted:   { buyerBlocked: ['view','add_to_cart','confirm_booking'], banner: { titleKey: 'listingBannerDeletedTitle',  bodyKey: 'listingBannerDeletedBody',  severity: 'destructive' } }
```

Resolver shape (D-14):
```
function resolveBlockedBuyerActions(state) { return LISTING_STATUS_POLICY[state]?.buyerBlocked ?? []; }
```
Note: v1.0 `resolveRestrictedFeatures` THROWS on unknown state; D-14 chose nullish-coalesce return for the listing variant (different rationale — listing state arrives from arbitrary client paths in Phase 9, defensive fallback preferred). CONTEXT decision is authoritative.

Export shape:
```
module.exports = { LISTING_STATUS_POLICY, resolveBlockedBuyerActions };
```

Test schema-equality lock (per 07-PATTERNS.md §10 adaptation notes):
```
const Car = require('../../src/models/Car');
const enumValues = new Set(Car.schema.path('status').enumValues);
const policyKeys = new Set(Object.keys(LISTING_STATUS_POLICY));
expect(policyKeys).toEqual(enumValues);   // {'active','suspended','archived','deleted'}
```
This is the foundational invariant — adding a new state to `Car.status` without updating `LISTING_STATUS_POLICY` (or vice versa) MUST break this test.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Module ↔ caller | Phase 9 enforcement code (cart-add, confirm-booking re-verify) will pass a `state` string from the database into `resolveBlockedBuyerActions`. Unknown state → empty list (defensive default per D-14). |
| Phase 11 translation key contract | The `banner.titleKey` / `banner.bodyKey` strings are an implicit contract with Phase 11's `translations.ts`. A typo here without a Phase 11 update would surface as a missing-translation warning at the buyer banner render. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03-01 | Tampering | schema drift between `Car.status` enum and `LISTING_STATUS_POLICY` keys | mitigate | Task 2 test 1 asserts set-equality between the two; CI fails on drift. |
| T-07-03-02 | Elevation of privilege | downstream consumer passes a string from untrusted input (e.g., URL param) to `resolveBlockedBuyerActions` and receives a "what is blocked" hint that could expose internal moderation taxonomy | accept | Function is pure-information; the returned blocked-action list does not grant any access. Phase 9 will use this purely server-side for its own re-verify decisions. |
| T-07-03-03 | Information disclosure | banner copy keys could be enumerated by a malicious client to infer the listing-moderation taxonomy | accept | The taxonomy (active/suspended/archived/deleted + 5 reason categories) is already user-visible by design (LBUY-01 banner shows the state to buyers). No additional disclosure surface. |
