`express-rate-limit` factory call (clone of v1.0 `src/moderation/rateLimit.js` lines 17–46):
- `windowMs: 15 * 60 * 1000` (15 minutes) — BYTE-IDENTICAL to v1.0
- `max: 30` — BYTE-IDENTICAL to v1.0
- `standardHeaders: true` — emits `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset` per RFC 6585
- `legacyHeaders: false` — no `X-RateLimit-*`
- `keyGenerator: (req) => { ... }` — THE LOAD-BEARING DIFF. Three tiers:
  - if `req.admin && req.admin.uid` → return `'listing-admin:' + req.admin.uid`
  - else if `req.admin && req.admin.email` → return `'listing-admin-email:' + req.admin.email`
  - else → return `'listing-unauthenticated'`
- `handler: (req, res) => { ... }` — BYTE-IDENTICAL to v1.0:
  - compute `resetTimeMs = req.rateLimit && req.rateLimit.resetTime ? req.rateLimit.resetTime - Date.now() : WINDOW_MS`
  - `retryAfter = Math.max(0, Math.ceil(resetTimeMs / 1000))`
  - `res.status(429).set('Retry-After', String(retryAfter)).json({ error: 'rate_limited', retryAfter })`

Exports (mirror v1.0 shape so the test file can import the constants for its 30-then-429 assertion):
- `module.exports = { listingModerationRateLimiter, WINDOW_MS, MAX_REQUESTS };`

D-04 separate-bucket invariant: the key prefix MUST be `listing-admin:` (and the email/unauthenticated fallbacks MUST carry the `listing-` prefix too). A regression that drops the prefix from ANY tier would make the listing limiter share state with the v1.0 user-mod limiter under that auth-degradation path — Plan 07-05's separate-bucket test will catch the happy-path tier, but the email/unauthenticated tiers are not directly tested; the grep acceptance below locks all three.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Middleware ↔ memory-store backend | express-rate-limit v8 default memory store is per-process. Single-instance Railway deploy currently (D-04 acknowledges memory-store sufficiency). Horizontal scale (deferred to v1.2+) would require a Redis store swap. |
| Limiter key ↔ admin identity | Key is derived from `req.admin.uid` (populated by `requireAdmin` upstream). If `requireAdmin` ever changes its req-shape contract, the keyGenerator's 3-tier fallback (uid → email → 'listing-unauthenticated') prevents single-bucket collapse but does NOT prevent per-admin-bucket loss. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04-01 | Denial of service | abusive admin (compromised credential) drains listing-moderation throughput | mitigate | 30 actions / 15 min / admin is the LSEC-03 budget; Plan 07-05's supertest proves the 31st returns 429. |
| T-07-04-02 | Denial of service | abusive admin drains user-moderation throughput via the listing endpoints (shared bucket attack) | mitigate | D-04 separate bucket — `listing-admin:` prefix isolates the listing-mod limiter from the v1.0 user-mod limiter. Acceptance grep on keyGenerator prefix locks this in CI. Plan 07-05 supertest exercises the live-chain proof. |
| T-07-04-03 | Spoofing | attacker rotates IPs to bypass the limiter | mitigate | Key is `req.admin.uid` (Firebase identity), not IP — IP rotation doesn't change the bucket. v1.0 Phase 2 Plan 02-06 Block 3 Test 2 established this; Phase 7 inherits the same property. |
| T-07-04-04 | Information disclosure | 429 response leaks `retryAfter` window timing, enabling a timing-oracle attack | accept | RFC 6585 `Retry-After` is the standardized response shape; the timing data is intentional and required for client backoff. No additional disclosure surface beyond what the standard mandates. |
| T-07-04-05 | Elevation of privilege | bypass of `requireAdmin` upstream means keyGenerator falls through to `listing-unauthenticated` bucket (all unauthenticated callers share one bucket) | accept | If `requireAdmin` is bypassed the request shouldn't have reached the limiter at all — defense-in-depth fallback ensures degraded operation (one shared bucket) rather than per-IP key-explosion DoS. Same disposition as v1.0 D-32. |
</threat_model>
