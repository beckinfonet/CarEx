<objective>
Fix CR-04: `CarDetailsScreen.tsx` line 124-125 early-returns the deep-link `apiClient.get('/api/cars/${carId}')` when `route.params.carData` is present. 6 of 7 navigation surfaces (HomeScreen, HomeScreenV2, MyListingsScreen, SearchResultsV2, FavoritesScreen, SellerListingsScreen) pre-populate carData — leaving `fetchedCar=null` for the admin, which means: (a) the `moderationBadge` prop to ListingModerationBottomSheet is `undefined` so the sheet always renders the 4-action active branch even on suspended/archived/deleted listings, (b) the admin status banner above the action area never renders, (c) the optimistic-flip handler at lines 489-499 is a no-op (`setFetchedCar((c) => c ? {...} : c)`), and (d) the rollback at line 517 is also a no-op.

Resolution: Change the fetch gate from `if (carId && !existingCar)` to `if (carId && (!existingCar || isAdmin))`. This guarantees `fetchedCar` populates with the Phase 9 D-07 `moderationBadge` payload for every admin entry path. The optimistic flip + rollback handlers then operate on a real `fetchedCar` object on all 7 surfaces. Non-admin viewers continue to use the fast-path (carData without network round-trip) — zero behavior change for them.

Purpose: Close the CR-04 BLOCKER from Phase 10's VERIFICATION.md. Without this fix, success criteria #1 and #3 remain PARTIAL — the admin moderation surface is silently inert on 6 of 7 navigation paths (>95% of admin entry points in practice). The fix is a single-predicate change plus one new regression test; it preserves the single-source-of-truth invariant (`fetchedCar.moderationBadge`) instead of introducing a parallel `moderationOverride` useState that would add drift risk.

Output: ~3 lines changed in CarDetailsScreen.tsx (1 predicate + 1 dep-array entry + 1 comment), ~1 new test (~30-50 lines) in CarDetailsScreen.admin.test.tsx mounting with carData populated and isAdmin=true.
</objective>
