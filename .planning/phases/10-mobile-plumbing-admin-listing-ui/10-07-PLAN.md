<objective>
Land three modal pieces for Phase 10's bottom-sheet workflow: (1) `ListingModerationReasonModal` for collecting reasonCategory + note on Suspend/Archive/Delete; (2) `ListingRestoreModal` for collecting an optional note on Restore; (3) an additive `keyboardType?: KeyboardTypeOptions` prop on the existing `TypedConfirmationModal` so the Delete escalation can render a normal keyboard (with spacebar) for typing the listing-title sentinel like "2018 Toyota Camry".

Purpose: LUI-02 (4 visually distinct actions) and LUI-03 (Restore swap) depend on these modals. D-04 sibling discipline (per CONTEXT) requires the new reason modal to embed the 5-value listing taxonomy directly (no cross-domain enum sharing with user-mod's 4-value `ReasonCategory`). Pitfall 3 mandates the `keyboardType` addition — without it, iOS users see the email-keyboard with no spacebar and physically cannot type the listing-title sentinel. The change is purely additive (default `'email-address'`) so existing user-mod call sites continue to render the email keyboard.

Output: Two new component files + one additive prop on an existing component + two co-located test files. Total ~250 LOC across modals; ~80 LOC of tests.
</objective>
