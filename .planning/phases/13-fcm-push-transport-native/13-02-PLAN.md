<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile client → device-token routes | Untrusted body; caller identity must come from the verified Bearer, never the payload. |
| Backend → FCM/APNs (lock screen) | Push payload crosses into a device's lock screen where PII would be exposed. |
| One device token → fan-out | A single poisoned/dead token must not affect delivery to others. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-02-01 | Information Disclosure | fcm.js push payload | mitigate | Render generic push_* copy via renderGenericPush; carry routing only in data.deeplink; PII-token assertion test (NPUSH-08/D-08b). |
| T-13-02-02 | Information Disclosure / TOCTOU | send-time listing state | mitigate | Upstream emit() already re-checks the hide-hook via plain Car.findById before fcm.send is reached; send does not bypass it (do not duplicate, do not skip). |
| T-13-02-03 | Elevation / Tampering (IDOR) | device-token routes | mitigate | uid = req.auth.uid only; DELETE filter scoped to caller uid; IDOR test proves 0 rows for another user's token. |
| T-13-02-04 | Denial of Service | fan-out on a poisoned token | mitigate | Per-token isolation: prune registration errors, bounded backoff on transient/429, never throw on one failure (NPUSH-05 test 1). |
| T-13-02-05 | Spoofing | FCM credential | accept | Reuse existing FIREBASE_SERVICE_ACCOUNT_JSON (cached OAuth via firebase-admin); A1 verification (FCM send scope) flagged in user_setup. |
</threat_model>
