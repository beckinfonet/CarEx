<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client → backend admin read endpoints | Admin's idToken in Authorization header is the sole authenticator on `/api/admin/users/search` and `/api/admin/moderation/{uid}/history`. ApiClient request interceptor (Plan 04-01) attaches it; backend must call verifyIdToken + getAdminStatus on every request. |
| mobile UI input → search query string | Typed search query is forwarded raw to backend; backend MUST validate (regex anchoring, length cap) to prevent ReDoS / unbounded-scan abuse |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Spoofing | searchUsers / getHistory called by non-admin caller | mitigate (server-side) | apiClient attaches Bearer; backend MUST verifyIdToken + getAdminStatus on every request. The mobile guardrail says the admin gate is UX only; authorization is server-authoritative (per CLAUDE.md "Auth enforcement"). Documented as backend contract — Plan 05-09 verification covers the cross-repo route handler audit; this plan does NOT trust mobile-side `isAdmin`. |
| T-05-03-02 | Information Disclosure | search response leaks PII for non-target users (e.g. private notes) | mitigate (server-side) | Backend route MUST project response to public-safe fields only; mobile types reflect that surface (no `setByAdminUid` exposed in SearchUserItem unless backend chooses to). Documented expectation; backend response shape dictates final type-narrowing. |
| T-05-03-03 | Denial of Service | unbounded `q` query causes backend ReDoS | mitigate (server-side) | Backend MUST cap `q` length (e.g. 100 chars), anchor regex, and reject empty queries. Mobile sends raw input; that's the contract. Documented for Plan 05-09 backend audit. |
| T-05-03-04 | Tampering | client passes a forged `cursor` to leak rows from another targetUid | mitigate (server-side) | Backend MUST sign cursors (HMAC) OR scope cursor to the requesting (admin, targetUid) tuple. Mobile passes the cursor opaque-as-received. Documented expectation. |
| T-05-03-05 | Information Disclosure | getHistory exposes admin emails as `adminEmail` field | accept | Admin emails are knowable by other admins by design (audit trail transparency); not a leak in admin-only context |
| T-05-03-06 | Tampering | mobile attaches a stale Bearer after admin logout | mitigate | Plan 04-04 already clears `currentIdTokenRef` first in logout; subsequent searchUsers call sends no Bearer; backend rejects 401. No new mitigation needed in this plan. |
</threat_model>
