<objective>
Close UAT Test 8 gap (severity: major) — implement automatic Firebase idToken refresh so admin actions continue to work after the 1-hour token TTL without forcing the user to log out and back in.

**Root cause (from UAT diagnosis):** `AuthContext` caches the idToken in `currentIdTokenRef` at sign-in and never refreshes it. `refreshUser()` only re-fetches the backend user record, NOT the Firebase token. After ~1hr idle, every auth-gated API call returns 401 until the user logs out + back in. Surfaced during UAT round 2 as a 401 on `searchUsers`; manual workaround (re-login) confirmed the diagnosis.

**Fix:** Two-pronged refresh strategy mirroring industry-standard Firebase token-refresh patterns:

1. **Proactive refresh:** AuthContext tracks `idTokenExpiresAt` (epoch ms). Before every protected call (or on a periodic check), if `remainingMs < 5 minutes`, refresh in the background. Implementation: a lightweight scheduler that runs the check on AppState `active` transitions (already wired) plus a one-shot self-rescheduling timer.

2. **Reactive refresh:** A new 401 response interceptor in `apiClient` (mirroring the existing 403 moderation interceptor) calls a registered `idTokenRefreshListener` (which AuthContext owns), waits for the refresh to complete, then retries the failed request ONCE with the new Authorization header. On a second 401, the interceptor gives up and triggers logout. Concurrent 401s share a single in-flight refresh promise (single-flight) so N parallel requests cause exactly ONE refresh.

3. **Token endpoint:** New `AuthService.refreshIdToken(refreshToken)` POSTs to `https://securetoken.googleapis.com/v1/token?key=${API_KEY}` with form-encoded body `grant_type=refresh_token&refresh_token=<token>`. Returns `{ id_token, refresh_token, expires_in }` (snake_case per Firebase REST contract). On HTTP 400 with `error.message === 'TOKEN_EXPIRED' | 'USER_DISABLED' | 'INVALID_REFRESH_TOKEN'`, the listener triggers logout.

4. **Persistence:** Sign-in / sign-up responses already include `refreshToken` and `expiresIn` (verified by reading existing `signIn` / `signUp` methods — they return `data` which contains `data.refreshToken` and `data.expiresIn` per Firebase REST docs). AuthService.saveToken is extended (or a new `saveAuthSession` helper added) to also persist `refreshToken` and `idTokenExpiresAt` to AsyncStorage. `loadStorageData` hydrates them on cold start.

Purpose: closes the second remaining major UAT gap; eliminates the "log out + back in every hour" workaround; restores end-to-end admin moderation reliability for sessions longer than the 1-hour Firebase token TTL.
Output: 5 modified files; pure axios + AsyncStorage (no new dependencies).

```typescript
  // --- Firebase token refresh (Plan 05-12 / UAT Test 8) ---
  //
  // Firebase Identity Toolkit issues idTokens with a 1-hour TTL. After expiry,
  // every protected backend call returns 401 until a new idToken is minted via
  // the securetoken endpoint using the long-lived refreshToken obtained at
  // sign-in. This method is the only mobile-side caller of that endpoint.
  //
  // Endpoint: POST https://securetoken.googleapis.com/v1/token?key=API_KEY
  // Content-Type: application/x-www-form-urlencoded
  // Body: grant_type=refresh_token&refresh_token=<token>
  // Returns SNAKE_CASE { id_token, refresh_token, expires_in, ... } —
  // normalized to camelCase here so the rest of AuthService stays consistent.
  //
  // Errors:
  //   - 400 INVALID_REFRESH_TOKEN / TOKEN_EXPIRED / USER_DISABLED → permanent
  //     failure. Caller (AuthContext) should logout the user.
  //   - Network error → transient. Caller may retry.
  refreshIdToken: async (refreshToken: string) => {
    try {
      const body = `grant_type=refresh_token&refresh_token=${encodeURIComponent(refreshToken)}`;
      const response = await axios.post(
        `https://securetoken.googleapis.com/v1/token?key=${API_KEY}`,
        body,
        { headers: { 'Content-Type': 'application/x-www-form-urlencoded' } },
      );
      // Normalize snake_case → camelCase to match signIn/signUp shape.
      return {
        idToken: response.data.id_token,
        refreshToken: response.data.refresh_token,
        expiresIn: response.data.expires_in,
      };
    } catch (error: any) {
      // Permanent-failure shape: rethrow with the Firebase error message so
      // AuthContext can branch on it (TOKEN_EXPIRED → logout vs network →
      // retry-later).
      throw error.response ? error.response.data.error : error;
    }
  },

  // --- Session persistence helpers (Plan 05-12) ---
  //
  // saveAuthSession extends the original saveToken (kept for back-compat) by
  // also persisting refreshToken + idTokenExpiresAt. Called by AuthContext
  // login/signup AND by the refresh path so the new idToken survives a cold
  // start.
  saveAuthSession: async (
    idToken: string,
    refreshToken: string | null,
    expiresInSeconds: number | string,
    userData: any,
  ) => {
    const seconds =
      typeof expiresInSeconds === 'string'
        ? parseInt(expiresInSeconds, 10)
        : expiresInSeconds;
    const expiresAt = Date.now() + (Number.isFinite(seconds) ? seconds : 3600) * 1000;
    await AsyncStorage.setItem('userToken', idToken);
    if (refreshToken) {
      await AsyncStorage.setItem('userRefreshToken', refreshToken);
    }
    await AsyncStorage.setItem('userIdTokenExpiresAt', String(expiresAt));
    if (userData !== undefined) {
      await AsyncStorage.setItem('userData', JSON.stringify(userData));
    }
  },

  getRefreshToken: async (): Promise<string | null> => {
    return await AsyncStorage.getItem('userRefreshToken');
  },

  getIdTokenExpiresAt: async (): Promise<number> => {
    const raw = await AsyncStorage.getItem('userIdTokenExpiresAt');
    return raw ? parseInt(raw, 10) : 0;
  },
```

    if (status !== 401 || !config) throw err;

    if (config._skipIdTokenRefresh || config._idTokenRefreshAttempted) {
      // Already tried once — trigger logout for permanent auth failure.
      if (config._idTokenRefreshAttempted && logoutTrigger) {
        try {
          await logoutTrigger();
        } catch (logoutErr) {
          console.error('Logout trigger failed after second 401', logoutErr);
        }
      }
      throw err;
    }

    if (!idTokenRefreshListener) {
      // No listener registered — fail open and propagate the original error.
      throw err;
    }

    let newToken: string | null = null;
    try {
      newToken = await idTokenRefreshListener();
    } catch (refreshErr) {
      console.error('idToken refresh listener failed', refreshErr);
      throw err;
    }
    if (!newToken) {
      // Listener already triggered logout (returned null on permanent
      // failure). Propagate the original 401 to the caller.
      throw err;
    }

    // Single retry with the new token. Mark the request as attempted so a
    // recurring 401 on the same request short-circuits next time around.
    config._idTokenRefreshAttempted = true;
    config.headers = config.headers ?? {};
    (config.headers as any).Authorization = `Bearer ${newToken}`;
    return apiClient(config);
  },
);

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile → Firebase securetoken | New external auth boundary. Pre-existing trust: idToken minted at sign-in. New trust: refreshToken can mint new idTokens until revoked. |
| AsyncStorage at rest | refreshToken now resides in device storage alongside the existing userToken. AsyncStorage on iOS/Android is sandboxed to the app but NOT encrypted-at-rest unless the device has lock-screen/biometrics enabled. |
| apiClient → backend | Existing — no boundary change. The 401 interceptor adds a single retry path that reissues the SAME request shape with a NEW Authorization header. Backend cannot distinguish a refreshed call from a fresh call. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-12-01 | Information Disclosure | refreshToken at rest in AsyncStorage could be extracted via device-rooted exfiltration or backup theft, granting attacker durable token-mint access | accept | Same risk profile as the existing idToken in AsyncStorage (Plan 04-04). Firebase Identity Toolkit recommends device-side storage for mobile apps; alternatives (Keychain on iOS, KeyStore on Android) require new native modules outside this milestone. Future hardening tracked in CONCERNS / deferred-items. |
| T-05-12-02 | Spoofing | A leaked refreshToken can mint idTokens until manually revoked in the Firebase console | mitigate | Listener triggers logout on TOKEN_EXPIRED / INVALID_REFRESH_TOKEN / USER_DISABLED — server-side revocation immediately ends the session. Admin can revoke via Firebase Console → Authentication → Revoke refresh tokens. Documented in PROJECT.md auth model section (deferred to future milestone). |
| T-05-12-03 | Denial of Service | Single-flight bug — if idTokenRefreshInFlightRef.current is set but the underlying promise rejects without clearing the ref, all subsequent refreshes deadlock waiting on the dead promise | mitigate | The listener's IIFE wraps refreshIdToken in try/finally; the finally clears idTokenRefreshInFlightRef. Test 13 explicitly verifies single-flight behavior. The proactive path mirrors the same pattern. |
| T-05-12-04 | Tampering | A 401 response interceptor that retries with a new token could allow an attacker to inject responses if Bearer tokens leak to a MITM | accept | All apiClient calls go to API_URL which is HTTPS in production (NSAllowsArbitraryLoads=false on iOS, cleartextTraffic=false on Android — both verified in CLAUDE.md). The 401 retry uses the same TLS channel as the original request. No new MITM surface. |
| T-05-12-05 | Elevation of Privilege | A second 401 after refresh could indicate token revocation MID-session — failing to logout would let the user appear authenticated while the server has cut them off | mitigate | The 401 interceptor's `_idTokenRefreshAttempted` flag triggers `logoutTrigger()` on the second 401, which routes through AuthContext.logout, clearing all refs + AsyncStorage. Test 14 covers the listener-side equivalent (TOKEN_EXPIRED on the refresh itself). |
| T-05-12-06 | Repudiation | A refresh that succeeds on the mobile but fails to persist to AsyncStorage (e.g. storage quota exceeded mid-call) leaves the in-memory refs ahead of disk — a cold start would replay the OLD token | mitigate | saveAuthSession is awaited before the listener returns the new token, so the persistence happens BEFORE the interceptor retries the failed request. If saveAuthSession throws, the listener's outer catch surfaces the error and the request 401s; AuthContext is no worse off than before the refresh. |
