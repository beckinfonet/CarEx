## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile client → POST /api/orders body | `item.providerSnapshot` in the request body must be treated as UNTRUSTED; handler now ignores it and resolves server-side. |
| server.js → Broker/LogisticsPartner/User collections | Server-authoritative lookup adds one Mongo read per unique provider per order; cheap at current scale. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Client-forged providerSnapshot | mitigate | Handler now REPLACES client-supplied providerSnapshot with server-resolved values. Test case "stale-from-client" → "Acme Brokerage" proves the client cannot influence the snapshot. |
| T-03-02 | Info disclosure | providerSnapshot leaking provider PII (email, firstName, lastName) to the buyer | accept | Buyer already sees provider contact info in the existing product flow (phone, telegram for booking). Adding email/name is consistent with product intent. Phase 2's DELETE provider-profile flow relies on this data surviving the delete — not leaking; preserving. |
| T-03-03 | Integrity | Snapshot goes stale between order creation and later reads | accept | snapshotAt field captures WHEN the snapshot was taken. Updates to the provider profile AFTER order creation do NOT update the snapshot — that's the entire point (Pitfall #3 / D-22). Buyer UIs show snapshot data as-of-creation, which is the correct anti-pattern-6 solution to cascading deletes. |
| T-03-04 | Denial of service | Handler does N extra Mongo reads per order (N = unique providers) | accept | Current app scale = 1-5 admins, low order volume; the additional 1-2 reads per create are negligible. If volume grows the handler can be revisited, but not in this milestone. |
| T-03-05 | Info disclosure | Warning log contains provider UID | accept | UIDs are not secrets; they appear in other logs and in buyer-visible URLs. Log level is `console.warn`, not `console.error`, so ops can filter. |
| T-03-06 | Tampering | server.js module.exports change breaks production startup | mitigate | `if (require.main === module)` preserves production behavior exactly; Railway still runs `node server.js` which triggers the listen(). Tests import the file without listening. This pattern is standard Node. |
</threat_model>
