```js
moderationStatus: {
  state: { type: String, enum: ['active', 'feature_limited', 'blocked_with_review', 'permanently_banned'], default: 'active', required: true },
  severity: { type: String, enum: ['none', 'feature_limited', 'blocked_with_review', 'permanently_banned'], default: 'none' },
  reasonCategory: { type: String, enum: ['spam', 'policy_violation', 'fraud', 'other'], default: null },
  note: { type: String, default: null, maxlength: 2000 },
  setByAdminUid: { type: String, default: null },
  setAt: { type: Date, default: null },
  restrictedFeatures: { type: [String], default: [] },
  lastActionId: { type: mongoose.Schema.Types.ObjectId, ref: 'ModerationAction', default: null },
}
```

Target indexes per D-13 (place AFTER `const userSchema = ...` definition, BEFORE `module.exports = mongoose.model(...)`):
```js
userSchema.index({ 'moderationStatus.state': 1 });
userSchema.index({ 'moderationStatus.state': 1, 'moderationStatus.reasonCategory': 1 });
```

Note: Because `reasonCategory` defaults to `null` but the enum array does not include `null`, Mongoose 9 treats `null` as "skip enum validation" (documented behavior). This is the intended semantic: reasonCategory is required when state !== 'active' (enforced by the moderation service in Phase 2), but nullable on active users. Do NOT add `null` to the enum array — it would allow admins to set reason to null on a suspended user.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mongoose application → MongoDB User document | moderationStatus subdoc now carries authorization-relevant state; incorrect writes here will leak moderation decisions or allow bypass. |
| Any future endpoint writing to user.moderationStatus | Only Phase 2's ModerationService should mutate this subdoc; Phase 1 just locks the shape. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | user.moderationStatus unauthorized write | mitigate | Schema enum on `state` prevents "banned"/"deleted"/"unknown" values sneaking in via malformed payloads. Phase 2 will add service-layer authorization around writes; this plan locks the data contract. |
| T-02-02 | Info disclosure | moderationStatus leaking to non-admin user endpoints | mitigate | By keeping the subdoc as a SEPARATE namespace (not merging into the existing `sellerStatus` etc. per Pitfall #10), downstream GET handlers can explicitly select-or-exclude moderationStatus. Phase 3 enforcement will do this — Plan 02's job is to keep the namespace clean so Phase 3 CAN. |
| T-02-03 | Tampering | reasonCategory bypassed via null default | accept | The `null` default is intentional — a user in state='active' has no reason. Phase 2's suspend handler must guarantee reasonCategory is non-null when state !== 'active' (enforced in service layer, not schema). Called out explicitly in the <interfaces> note so the executor doesn't "fix" it. |
| T-02-04 | Tampering | note field accepting arbitrary HTML/markdown | accept | Stored as plain String; React Native's default text rendering is safe (no HTML interpretation — Pitfall 16.3). maxlength 2000 bounds blast radius. Sanitization is a UI concern; schema does its part by storing text, not HTML. |
| T-02-05 | Info disclosure | Indexes exposing query patterns | accept | Indexes on moderationStatus.state + reasonCategory are required for admin list queries (D-13, Phase 5). Index existence is not a secret; the data they index is protected by the Phase 2 requireAdmin middleware at the query layer. |
