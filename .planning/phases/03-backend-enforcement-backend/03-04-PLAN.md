<objective>
Extract a new shared service `../backend-services/carEx-services/src/payments/confirmBooking.js` that absorbs Stripe PaymentIntent retrieval, buyer/provider/seller re-verification, car `listingStatus` flip, and `ServiceOrder` row creation into a single Mongoose transaction. Rewrite the POST /api/payments/confirm-booking handler in `server.js` to delegate to this service and map thrown errors to 409 `provider_suspended`.

Purpose: Closes ENF-03 (ROADMAP Success Criterion #3) — the TOCTOU race where an admin suspends a provider between `create-payment-intent` and `confirm-booking`. The re-check inside the transaction, paired with refund-first-throw-second, guarantees the buyer is never charged for a booking that doesn't complete.
Output: One new service module + modified POST /api/payments/confirm-booking route handler.

Ordering rationale (D-11): Stripe API is external to Mongo. If we throw-first-refund-second and Stripe's refund API fails, the transaction aborts but the charge persists — the buyer loses money. Refund-first-throw-second means worst case is "refund succeeded, transaction aborted" (safe) OR "refund failed, error surfaced with refundFailed=true flag for ops reconciliation" (loud). Comment on the handler makes this explicit per CONTEXT.md §Specifics.

Depends on Plan 03-03 because both plans touch server.js's POST /api/payments/confirm-booking. Plan 03-03 adds the attachAuthIfPresent + requireNotSuspended('create_order') middleware chain; Plan 03-04 replaces the handler body. Serializing avoids merge conflict on the same route block.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client body → confirmBooking payload | `paymentIntentId`, `carId`, `buyerUid`, `items[]` are all attacker-controlled. Re-verification inside the transaction is authoritative. |
| Stripe API → server | External system; not in Mongo transaction; refund call happens before throw to prevent lost-money state. |
| Mongo transaction boundary | Buyer/provider/seller re-checks + car flip + ServiceOrder create MUST all be inside the same session. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04-01 | T (Tampering / TOCTOU) | confirm-booking provider race | mitigate | Re-check inside the transaction reads each party's current moderationStatus and role status with includeAllUsers bypass. Snapshot isolation (Phase 2 D-23) serializes with admin.suspend: either admin commits first and we refund, or we commit first and admin's suspend succeeds without rolling back this order. D-13 acceptance. |
| T-03-04-02 | D (DoS / Lost funds) | Stripe refund failure | mitigate | refundThenThrow catches refund errors, sets `refundFailed = true`, surfaces in 409 response so ops reconcile manually. Log via `console.error`. Documented in D-11. |
