<objective>
Replace the body of `POST /api/orders` in `server.js` with a 410 Gone response directing callers to `POST /api/payments/confirm-booking` (per D-12). The endpoint remains registered so stale mobile builds still calling it fail loudly and visibly rather than silently succeeding against a code path that no longer does the right thing.

Purpose: Removes the two-step flow risk. Before Phase 3, the mobile app called `create-payment-intent → confirm-booking → POST /api/orders` as three separate HTTP calls, which opened a window where an admin could suspend the provider between confirm-booking and order creation. After Plan 03-04 absorbs order creation INTO confirm-booking's transaction, the standalone `POST /api/orders` handler is dead code with a dangerous ability to create orders without provider re-check. Replacing it with 410 Gone closes the window unambiguously.

Depends on Plan 03-04 because 03-04 migrates the order-creation logic into `src/payments/confirmBooking.js`. Only after that migration is POST /api/orders safe to neutralize. Sequencing also avoids a merge conflict on adjacent lines in server.js (1113-1244 range).

Scope: Route body replacement ONLY. Route entry stays registered (removal is tracked in Deferred Ideas — happens after Phase 4 mobile stops calling it and a grace period passes).
</objective>
