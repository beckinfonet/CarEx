Key invariants the executor MUST preserve:
- Bypass flag `.setOptions({ includeAllListingStatuses: true })` — without it, the Plan 02 hide hook returns null for non-active listings and the gate silently passes through to Stripe (Pitfall 1 variant — same root cause as the in-txn hide hook pitfall, just at the gate boundary).
- `.select('status moderationReason')` — narrow read, never load the full Car doc here (avoids PII over-fetch + extra bandwidth + accidentally leaking the doc to logs).
- `.lean()` — plain JS object, no Mongoose hydration cost since we only read status + moderationReason.
- The 409 body MUST exactly match D-11's create-payment-intent variant: `{ error: 'listing_not_available', listingStatus, reasonCategory, banner }` — NO `refundId` or `refundFailed` fields (those only appear at the confirm-booking 409 variant per D-11).
- The 404 fallback for malformed carId returns `{ error: 'car_not_found' }` — matches the existing confirm-booking 404 error code at server.js:1078 for consistency across both payment endpoints (Shared Pattern S-5 + Pitfall 6).
- If `Car.findById` returns `null` (carId doesn't exist), the gate FALLS THROUGH to the existing Stripe call. This is intentional: Stripe will accept the payment intent (Stripe doesn't validate carId), and confirm-booking's existing `car_not_found` check will catch it downstream. Alternative behavior (return 404 from the gate) is acceptable IF the executor finds an existing test that depends on the fall-through behavior — verify via grep on existing tests before changing.
