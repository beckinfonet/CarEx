<objective>
LENF-02: Modify `GET /api/cars/:id` so that a SINGLE endpoint branches on
caller identity to serve admins the full Car document + a `moderationBadge`
field while non-admins (anonymous or authenticated buyer) viewing a non-active
listing get a PII-minimized thin payload with the locked D-05 allowlist plus
a `banner` block pulled from `LISTING_STATUS_POLICY[status].banner`.

Purpose: deliver ROADMAP Criterion #2 ("Listing-detail GET for a
suspended/archived/deleted listing returns a thin payload (status + reason
category only, no seller PII or moderation notes) to non-admin viewers;
admin viewers receive the full document plus status badge") with one handler
modification. No new admin-only endpoint (D-08). Non-admin viewers of ACTIVE
listings see the existing response shape unchanged.

A1 mapping decision (RESEARCH Open Question 1): The `moderationBadge` admin
field uses `reasonCategory = car.moderationReason` (the 5-value enum from
Phase 7 D-14a) and `moderationReason = car.moderationNote` (the free-text
admin note). Rationale: D-05 thin payload's `reasonCategory` and D-07's
`reasonCategory` both consistently mean the enum value; D-07's second
`moderationReason` field can only mean the free-text note (the only other
admin-relevant field on the schema). This mapping is codified in the response
shape and locked by an acceptance criterion that asserts the badge contains
both fields with the expected source-of-truth values.

Output: modified `server.js` GET /api/cars/:id handler + GREEN
`listingDetailStatusAware.test.js`.
