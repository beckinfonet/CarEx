# Phase 6 — UI Design Contract

> Visual and interaction contract for the affected-user mobile UX (React Native). Generated by gsd-ui-researcher, verified by gsd-ui-checker. **Scope is Half A only** (AFF-01..04); QUAL-01..03 merge-gate quality checks have no visual deliverables beyond ensuring all new strings land in `translations.ts`.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (React Native 0.83 — shadcn not applicable on native) |
| Preset | not applicable |
| Component library | stock `react-native` primitives (`View`, `Text`, `Modal`, `Pressable`, `TouchableOpacity`, `Alert`, `Linking`, `LayoutAnimation`). **No new UI libraries** per CLAUDE.md ("Don't introduce new state-management or networking libs for this milestone"). |

## Spacing Scale

Declared values (all multiples of 4 — reuse existing `SIZES` tokens from Phase 5):

| Token | Value | `SIZES` key | Usage in Phase 6 |
|-------|-------|-------------|------------------|
| xs | 4px | `spacingXs` | Icon-to-text gaps in banner row; reason-chip inner padding |
| sm | 8px | `spacingSm` | Banner left-accent-to-content gap; overlay card icon-to-title gap; mailto button icon gap |
| md | 16px | `spacingMd` (= `padding`) | Banner horizontal padding; overlay card interior padding; default vertical rhythm |
| lg | 24px | `spacingLg` | Overlay card outer margin from screen edge; vertical gap between overlay card and nearest UI |
| xl | 32px | `spacingXl` | Overlay card top offset when banner is present (banner height + `spacingXl`) |
| 2xl | 48px | `spacing2xl` | Not used this phase |

Exceptions:
- **Banner left-accent bar width = 4px** (same as `spacingXs`, functions as a visual accent not a spacing token — matches Phase 5 `historyCard.borderLeftWidth`)
- **Minimum touch target = 44px** (`SIZES.minTapTarget`) on the Appeal CTA, "Copy email" Alert button, and the overlay card's primary CTA — iOS HIG requirement
- **Banner collapsed min-height = 64px** (enough for 2 lines of 14px body at 20px lineHeight plus 16px vertical padding = 24+16+16+8 safe-area); **expanded max-height = uncapped** (auto-grow; full note rendered with `numberOfLines={undefined}`)

- Line 2 row: note Text (`TYPOGRAPHY.body`, `color: COLORS.textPrimary`, `marginTop: SIZES.spacingXs`, `numberOfLines: collapsed ? 2 : undefined`)

Label: `TYPOGRAPHY.bodyStrong`, `color: '#FFFFFF'`.

**Interaction contract:**
- Tapping the **line 2 note area** (Pressable wrapper) toggles local `expanded` state
  - Tap target area: line 2 Text container (NOT the whole banner — tapping line 1 or the appeal CTA must not toggle)
  - When `expanded === true` and `note` renders >2 lines, the banner auto-grows (no fixed maxHeight)
  - `expanded` resets to `false` on navigation away (use `useFocusEffect` from `@react-navigation/native`) — per D-02 "collapses on navigation away"
  - No AsyncStorage persistence — session only
- Tapping the **Appeal CTA** calls the mailto handler (see §Appeal CTA behavior below); button is the only interactive surface besides the note expander
- Banner is **non-dismissable** — no close button, no swipe gesture, no dismiss state ever

**Expand animation (Claude's Discretion resolved):**
- Use `LayoutAnimation.configureNext(LayoutAnimation.Presets.easeInEaseOut)` **immediately before** `setExpanded(...)`. Enables native layout transitions without importing `react-native-reanimated` (which is installed but heavier ceremony for a single expand/collapse). On Android, add `UIManager.setLayoutAnimationEnabledExperimental?.(true)` in a module-top-level effect at App.tsx startup.
- Rationale over fade/slide alternatives: LayoutAnimation preserves scroll position, respects OS accessibility "Reduce Motion" automatically, and requires zero state timing logic. Banner height change is the physical affordance; additional fade/slide is cosmetic noise.

**Accessibility:**
- Root View: `accessible={true}`, `accessibilityRole="alert"`, `accessibilityLiveRegion="polite"`, `accessibilityLabel={${severityTitle}. ${reasonCategoryLabel}. ${note || ''}}`
- Note Pressable: `accessibilityRole="button"`, `accessibilityLabel={expanded ? t.collapseNote : t.expandNote}`, `accessibilityHint` in RU/EN for "tap to show full reason"
- Appeal CTA: `accessibilityRole="button"`, `accessibilityLabel={t.appealCta}` (i.e. the button's visible label — "Обжаловать" / "Appeal")

- Note expand area: `user-status-banner-note`

### 3. GatedScreenWrapper (integration seam — decision below)

**Decision (Claude's Discretion resolved): Component wrapper, NOT hook.**

Rationale:
- `<GatedScreenWrapper capability="create_listing"><SellCarScreenBody /></GatedScreenWrapper>` is a single-file diff per screen (1 import + 2 JSX lines wrapping the existing top-level View)
- A `useCapabilityGate(capability)` hook would force each of the 4 gated screens to render the overlay themselves (or a second helper component), duplicating the pointer-events + dim-layer plumbing 4 times
- The component-wrapper API composes cleanly over the existing `SafeAreaView`/`ScrollView` in each screen without caller-side conditional-render branching

**File:** `src/components/moderation/GatedScreenWrapper.tsx` (NEW)

**API:**
```ts
interface GatedScreenWrapperProps {
  capability: CapabilityKey;
  children: React.ReactNode;  // the screen's existing body
}
```

**Implementation sketch (for UI-SPEC precision, not full code):**
```tsx
export const GatedScreenWrapper: React.FC<GatedScreenWrapperProps> = ({ capability, children }) => {
  const { user } = useAuth();
  const state = user?.moderationStatus?.state ?? 'active';
  const restricted = user?.moderationStatus?.restrictedFeatures ?? [];
  const isGated = state !== 'active' && restricted.includes(capability);

  if (!isGated) return <>{children}</>;

  return (
    <View style={{ flex: 1 }}>
      <View style={{ flex: 1 }} pointerEvents="none">
        {children}
      </View>
      <FeatureGateOverlay capability={capability} />
    </View>
  );
};
```

The check uses denormalized `restrictedFeatures` from Phase 1 DATA-01 (the backend's STATUS_POLICY capability map is the authoritative source; mobile reads the already-resolved array). This matches Phase 3 Plan 03-02's pattern (mobile gate reads the same denormalized array the backend 403 path consults).

### 4. CarDetails gating — targeted CTA replacement, NOT full overlay

**Decision (Claude's Discretion resolved):** On `CarDetailsScreen`, gate **only the "Contact seller" CTA surface**, NOT the entire screen.

Rationale:
- Browsing a car listing has buyer-side value even for moderated users (read-only access to see what they're missing, matches D-04's context-preservation philosophy)
- Gating the whole screen would hide public product data that any anonymous visitor can see — disproportionate restriction
- The backend enforces the actual denial (Phase 3 ENF-01); the overlay here is informational UX for the CTA the user would otherwise tap

**Visual treatment:**
- Leave the CarDetailsScreen body untouched
- Wrap the "Contact seller" button (exact JSX node identified during implementation — planner reads `src/screens/CarDetailsScreen.tsx` to locate the CTA row) with a lightweight `GatedButtonGate` — sibling component that renders a DIM + DISABLED button (outline style, `opacity: 0.4`, `pointerEvents="none"`) plus a tiny helper line below: "Connection restricted — tap for details" in severity fg color
- Tapping the helper line opens a **small modal** (reusing RN `Modal` with `animationType="fade"`) that renders the same `FeatureGateOverlay` card verbatim — but centered as a modal, not as a full-screen dim
- Keeps the overlay copy source-of-truth unified (same translation keys as the full-screen variant) while avoiding a full-screen takeover on a browse screen

## Clipboard Decision (D-08 resolution)

**Decision:** **Use no clipboard library at all for Phase 6.** Replace the "Copy email" button with the actual text displayed inline.

**Rationale:**
- `package.json` audit (lines 16-34) shows **neither** `@react-native-clipboard/clipboard` nor any clipboard package is installed
- RN 0.83 **removed the legacy built-in `Clipboard` module** (it was extracted to the community package years ago). Importing `Clipboard from 'react-native'` is undefined at runtime
- Adding `@react-native-clipboard/clipboard` violates CLAUDE.md ("no new networking or state-management libs") — the spirit extends to new native modules that require pod install + Android gradle refresh on a release-prep codebase
- D-08 "planner chooses based on what's already in the project" therefore resolves to: **neither path**

**Revised D-08 behavior:**
- Alert title: `appealNoMailTitle` (unchanged — see Copywriting Contract)
- Alert body: `appealNoMailBody` (unchanged — already includes the email address AND the UID in the visible Alert text, so the user can manually tap-and-hold to select+copy on both iOS and Android native text-in-Alert surfaces)
- Alert buttons: `{ text: t.appealOk, style: 'default' }` (single "OK" button — no "Copy email" button, no "Cancel" needed since Alert.alert's default native dismiss affordance is already present)

This means the `appealCopyEmail` / `appealCopied` / `appealCancel` translation keys listed in the Copywriting Contract above are NO LONGER NEEDED. They are struck from the final key count.

- Reason-category labels (`reasonSpam` etc.) — ALREADY EXIST in translations.ts from Phase 5; reused, not added

## Appeal CTA behavior (consolidated flow)

```
1. User taps Appeal CTA in UserStatusBanner (only visible when state === 'blocked_with_review')
2. Build mailto URL:
   const subject = `CarEx moderation appeal — ${user.localId}`;
   const body = [
     `User ID: ${user.localId}`,
     `Reason category: ${reasonCategoryLabel}`,  // localized
     `Suspended: ${user.moderationStatus.setAt}`,  // ISO8601
     '',
     t.appealPlaceholder,  // "[Your message here]" / "[Напишите ваше сообщение здесь]"
   ].join('\n');
   const url = `mailto:support@carexmarket.com?subject=${encodeURIComponent(subject)}&body=${encodeURIComponent(body)}`;
3. Call Linking.openURL(url).catch(() => {
     Alert.alert(
       t.appealNoMailTitle,
       t.appealNoMailBody.replace('{uid}', user.localId),
       [{ text: t.appealOk, style: 'default' }]
     );
   });
```

- `canOpenURL` is **not called** (matches CarEx convention at `CarDetailsScreen.tsx:220` — "Avoid canOpenURL - it returns false on some Android devices even when the target app is installed")
- Catch fallback runs on true no-mail-client case OR on any other openURL rejection; the Alert body surfaces the email + UID so the user can proceed manually
