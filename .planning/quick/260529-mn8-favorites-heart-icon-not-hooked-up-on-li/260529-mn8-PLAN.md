<objective>
**Mode:** quick

The heart icon on listing cards (BigFeedCard, SmallFeedCard — rendered in HomeScreenV2 and SearchResultsV2) does nothing when tapped. The cards are correctly built with `faved` + `onToggleFav` props, but the parent screens pass `onToggleFav={() => {}}` — a no-op — and `faved={!!item.faved}` reads a field that never exists on listing items.

**Goal:** Wire the heart icon to a working, persistent favorites toggle. Make CarDetailsScreen, HomeScreenV2, SearchResultsV2, and FavoritesScreen share a single source of truth.

**Output:** A small `FavoritesContext` (mirroring `CartContext` shape) that wraps the existing AsyncStorage key `favorites`, plus 3-line edits in 4 screens.
</objective>

<scouting_summary>

### Findings

1. **Heart icons live in `BigFeedCard.tsx` and `SmallFeedCard.tsx`**, NOT `CarCard.tsx` (the user's mention of "the listings page" refers to HomeScreenV2 + SearchResultsV2, which use the v2 feed cards — confirmed via MEMORY.md "HomeScreen v2 shipped 2026-05-28").
2. **The cards are correctly designed.** Both expose `faved: boolean` and `onToggleFav: (car) => void` props (SmallFeedCard.tsx:18-24, BigFeedCard.tsx:33-40), and both render the heart with `color/fill` based on `faved`. Existing component tests cover the toggle interaction (`src/components/home/v2/__tests__/SmallFeedCard.test.tsx`, `BigFeedCard.test.tsx`).
3. **The bug is in the parents.** `HomeScreenV2.tsx:236-237` and `SearchResultsV2.tsx:170-171` both pass `onToggleFav={() => {}}` and `faved={!!item.faved}`. There is no `item.faved` field on listings, so cards always render unfilled, and tapping them does nothing.
4. **CarDetailsScreen ALREADY has a working pattern** (`src/screens/CarDetailsScreen.tsx:187-215`) that reads/writes AsyncStorage key `'favorites'` — a JSON-serialized array of carIds. `FavoritesScreen.tsx:23-59` reads the same key. So infrastructure exists; only the v2 cards were never wired in.
5. **No backend favorites endpoint exists.** `grep -i 'favorit' src/services/AuthService.ts` returns zero matches. Local-only AsyncStorage is the right call per constraints.
6. **No FavoritesContext exists.** The current pattern is duplicated screen-local state (CarDetailsScreen + FavoritesScreen each call AsyncStorage directly). Adding a thin context unifies the three call sites and gives HomeScreenV2/SearchResultsV2 reactive `faved` values without per-card AsyncStorage roundtrips.
7. **FavoritesScreen.tsx:19-21 only fetches on mount**, so favorites added on HomeScreenV2 won't appear if the user navigates straight to FavoritesScreen without remounting it. Fixing this is part of the same context migration — `useFavorites().favoriteIds` is reactive, so the screen will re-render when the set changes. (No need for `useFocusEffect` once the context owns the state.)

### Minimal wiring needed

- Create `src/context/FavoritesContext.tsx` — mirror `CartContext.tsx` shape exactly: provider hydrates from `AsyncStorage.getItem('favorites')` on mount, exposes `{ favoriteIds: Set<string>, isFavorite(id), toggleFavorite(id) }`, persists on every mutation, and resets state when `user?.localId` changes (same `prevUidRef` pattern at `CartContext.tsx:51-64`).
- Wrap app: `<FavoritesProvider>` under `<AuthProvider>` (needs `useAuth`) and outside `<NavigationContainer>` so all screens can consume.
- HomeScreenV2 + SearchResultsV2: import `useFavorites`, replace `faved={!!item.faved}` with `faved={isFavorite(item.id)}` and `onToggleFav={() => {}}` with `onToggleFav={(car) => toggleFavorite(car.id)}`.
- CarDetailsScreen: replace local `isFavorite` state + `checkFavoriteStatus`/`toggleFavorite` functions with `useFavorites()`. The two ASCII strings `'favorites'` for the AsyncStorage key already match the context's key — no migration needed; existing data persists.
- FavoritesScreen: replace `useEffect(fetchFavorites, [])` trigger with reactive dependency on `favoriteIds` (from context), so navigating in after favoriting on HomeScreenV2 shows the new item.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| device storage → app memory | AsyncStorage 'favorites' contents are read into the FavoritesProvider state on mount; an attacker with device access could write arbitrary JSON to this key. |
| user A session → user B session | Same device, sequential logins; per-user reset must fire on `user.localId` transition. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-MN8-01 | Tampering | AsyncStorage key 'favorites' | accept | Local-only UX state with no privilege; corrupted JSON triggers a try/catch and starts the user with an empty Set. No PII. |
| T-MN8-02 | Information Disclosure | Cross-user leak on shared device | mitigate | `prevUidRef` + `useEffect([user?.localId])` resets the in-memory Set when localId transitions (mirrors CartContext.tsx:57-64). NOTE: AsyncStorage data is NOT cleared on logout — it persists per-device, but the next user starts fresh in-memory and overwrites on first toggle. This matches CartContext behavior; an explicit AsyncStorage.removeItem on logout is out of scope for a quick fix. |
| T-MN8-03 | Denial of Service | Large favorite count | accept | Set serialized as JSON; ~30 bytes per id × any reasonable number of favorites stays well under AsyncStorage's per-key limit. No mitigation needed. |
</threat_model>
