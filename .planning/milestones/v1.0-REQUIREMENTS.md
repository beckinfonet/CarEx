## v1 Requirements — Outcomes

### Security

- [x] **SEC-01** — Validated. Backend verifies caller's Firebase ID token via `firebase-admin.auth().verifyIdToken()` on every admin route. Implemented in Phase 1 (01-05). Verified via 06-SECURITY.md Section (a) + grep evidence.
- [x] **SEC-02** — Validated. Every admin moderation endpoint enforces admin-only access server-side via `requireAdmin` middleware. Mobile `isAdmin` is never trusted. Verified via 06-SECURITY.md Section (a-b).
- [x] **SEC-03** — Validated. Backend rejects self-moderation + last-admin attempts (returns `400 cannot_moderate_self` / `400 last_admin_protected`). Phase 2 (02-03/04).
- [x] **SEC-04** — Validated. Admin moderation endpoints rate-limited via `express-rate-limit` (30 actions / 15 min / admin → 429). Phase 2 (02-06).

### Data Model

- [x] **DATA-01** — Validated. `User.moderationStatus` subdoc with all required fields shipped. Phase 1 (01-02 + 01-06 backfill).
- [x] **DATA-02** — Validated. `ModerationAction` collection is append-only at the application layer (6 pre-hooks reject update*/delete*); indexed on `{targetUid:1, createdAt:-1}` and `{adminUid:1, createdAt:-1}`. Phase 1 (01-01). Verified via 06-SECURITY.md Section (d).
- [x] **DATA-03** — Validated. `Order.providerSnapshot` denormalized; backfill migration ran. Phase 1 (01-03 + 01-06).
- [x] **DATA-04** — Validated. Capability map at `src/moderation/capabilities.js` with `STATUS_POLICY` + `resolveRestrictedFeatures`. Phase 1 (01-04).

### Admin Actions

- [x] **ADMIN-01** — Validated. Suspend with severity + reason category + optional note in single Mongoose transaction. Phase 2.
- [x] **ADMIN-02** — Validated. Unsuspend returns user to `active`; listings re-surface via read-time filter (no mutation). Phase 2.
- [x] **ADMIN-03** — Validated. Revoke role downgrades user; provider profile preserved for history. Phase 2.
- [x] **ADMIN-04** — Validated. Hard-delete provider profile; past orders intact via `Order.providerSnapshot`. Phase 2.
- [x] **ADMIN-05** — Validated. Edit provider profile fields; audit row contains `fieldDiff`. Phase 2.

### Backend Enforcement

- [x] **ENF-01** — Validated. `requireNotSuspended` middleware on 5 ROADMAP-named routes returns `403 account_suspended`. Phase 3 (03-02/03).
- [x] **ENF-02** — Validated. Mongoose `pre(/^find/)` hide hooks; admin opts in via `setOptions({ includeAllUsers: true })`. Phase 3 (03-01).
- [x] **ENF-03** — Validated. confirm-booking re-verifies provider status inside transaction; refund-first-throw-second on suspend mid-window. Phase 3 (03-04).
- [x] **ENF-04** — Validated. No `listing.active` mutation on suspend; visibility computed at read time. Phase 3 (03-01).

### Admin UI

- [x] **UI-01** — Validated. Per-row quick-action menu on AdminManagementScreen (Suspend / Unsuspend / Revoke / Delete / Edit) + dual-role delete contract (TWO rows for users with both broker + logistics). Phase 5 (05-06, 05-09).
- [x] **UI-02** — Validated. AdminModerationScreen with email/UID search, role/state filters, paginated results. Backed by GET /api/admin/users/search (cross-repo Plan 05-0b — verified live 2026-04-30, 18 jest tests green).
- [x] **UI-03** — Validated. Per-user detail panel with full moderation history. Backed by GET /api/admin/moderation/:targetUid/history (cross-repo Plan 05-0a — verified live 2026-04-30, 9 jest tests green).
- [x] **UI-04** — Validated. Unsuspend from history view; new audit entry appended; history never edited.
